# ⛩️ 羅生門：開発・統治プロセス（DEVELOPMENT_WORKFLOW）

本文書は、主殿（人間）と四役（AI）が、「羅生門 RPG」の再建において遵守すべき**絶対的な黄金律**を定めたものである。手法の変更時には必ず本文書を改訂し、歴史の連続性を保たねばならない。

---

## 1. 黄金の五段階サイクル（The Five Sacred Steps）

いかなる改修も、以下の段階を一つずつ、主殿の明示的な承認を「鍵」として進めなければならない。

### ① アルゴリズムレビュー（Logic Review）
- **目的**: 実装（コード）という不純物が混じる前に、解決の「論理」を主殿と合意する。
- **行動**: 過去のログと現状の解析に基づき、状態遷移図や論理フローを提案する。
- **承認**: 主殿の合意を得るまで、コードの書き換え（実装）を厳禁する。

### ② 四役会議・事前コードレビュー（Pre-Code Review）
- **目的**: 具現化されるコードの「一行一魂」を全方位で検分する。
- **行動**: 変更予定の差分（micro-diff）を逐一抽出し、設計・実装・監察・計量の四役による多角的な判定記録を主殿に報告する。
- **記録**: `git diff` 形式、または変更箇所の詳細なスニペットを用いる。

### ③ 承認の儀（Final Review Approval）
- **目的**: 設計と実装案の一致を主殿が審判する。
- **行動**: 事前コードレビューの結果に対し、主殿の最終的なゴーサインを仰ぐ。

### ④ 実装とロジック検証（Implementation & Test-First）
- **目的**: 承認された理を、数学的・論理的な正確さをもって具現化する。
- **行動**: 実装と同時にテストコード（Vitest 等）を走らせる。MVC 分離を徹底し、副作用を 0 に近づける。
- **禁忌**: ブラウザエージェントによる自律的な確認は原則禁止。異常発生時のログ収集・現場検証にのみ受動的に用いる。

### ⑤ 最終審判（Human Verification）
- **目的**: 平安の「体験」に主殿が魂を吹き込む。
- **行動**: 主殿自らがブラウザで動作を確認する。AI はその際に検出された異常系ログの解析と原因究明に徹する。
- **奉納（Release Strategy）**: 
    - 検証中のコードは **(unstable)** タグ、または「リリースタグ無し」でコミットし、履歴の透明性を保つ。
    - 主殿による「最終審判」をクリアし、安定動作が確認された理のみを **(stable)** としてリリースタグを打ち、安定版へと昇格させる。

---

## 2. 四役の役割と責任（The Four Pillars）

- **設計官（Architect）**: 「アルゴリズムレビュー」の主導。MVC の法、データの器（Contract）の整合性を護る。
- **実装官（Implementer）**: 「一行一魂」の体現。JSDoc の付与、副作用の排除、テストファーストの執行を担う。
- **監察官（Auditor）**: 「微細な塵」の排除。Lint、Diff、セキュリティ、および承認なき実装の弾劾を担う。
- **計量官（Analyst）**: 「都の平穏」の俯瞰。KPI 管理、複雑度、そして主殿への戦略的アジャストの具進を担う。

---

## 3. 聖典の改訂に関する掟（Amendment Protocol）

開発手法を破棄、あるいは大規模に改訂する場合は、必ず本文書自体を先に修正し、過去の自分たちの「智」に対する不徳を認め、新しき法がなぜ旧きを上回るのかを論理的に説明しなければならない。
