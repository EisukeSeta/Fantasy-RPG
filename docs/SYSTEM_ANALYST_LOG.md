## 🗺️ モジュール機能・役割診断（System Landscape Analysis）

### 1. [中枢] GameContext.jsx
- **役割**: 都の全事象（State）を司る根源。
- **実装特長**: `useState` の集合体。副作用を `useEffect` で統合的に監視する「監視塔」の性格を持つ。
- **受け渡しの特徴**: Context API によるブロードキャスト。バケツリレーを排除しているが、大規模更新時の「不必要な震え（再レンダリング）」が潜在的リスク。
- **戦略的評価**: 現在は安定しているが、新章追加時には「探索ステート」と「戦闘ステート」の Provider 分離が必須となる。

### 2. [戦場] useCombat.js (Hooks)
- **役割**: 現場（UI）と理（計算）の調停者。
- **実装特長**: 非同期タイマーによるターン制御、演出ステート（VFX）のトリガー。
- **受け渡しの特徴**: `combat.js` の計算値を `gameState` へ流し込み、UI がそれを検知して表示を切り替える「ステート駆動」型。
- **戦略的評価**: 演出（Sound/VFX）の呼び出しがインラインで記述されており、結合度が極めて高い。第3段階の「オーケストレーター」導入により、この結合度を 50% 低減させる必要がある。

### 3. [理] logic/combat.js (Pure Logic)
- **役割**: 都における「絶対的な計算式」。
- **実装特長**: 副作用を持たない純粋関数。JSDoc による入出力定義が最も進んでいる領域。
- **受け渡しの特徴**: `(Player, Enemy, Action) -> Result` という明快な契約。
- **戦略的評価**: 都の理想形。全ての機能はこの設計思想に倣うべきである。

### 4. [伝聞] DialogContext.jsx / UI Components
- **役割**: 物語と主殿を繋ぐ言葉（Dialog）の制御。
- **実装特長**: ページング機能を持つダイアログ制御、選択肢による分岐管理。
- **受け渡しの特徴**: `setActiveDialog` を通じて動的に物語を注入。
- **戦略的評価**: 現在、戦闘終了後の「敗北ダイアログ」が `isBusy` フラグと競合する。演出順序の厳格なキュー管理（Queueing）が次なる戦略目標。
