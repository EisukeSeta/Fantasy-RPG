# ⛩️ 都の再開発計画書 (Modularization & Testing Strategy)

## 1. 現状分析と課題
- **App.jsx の肥大化**: 現在 600 行を超え、UIロジックと状態管理が混在。
- **トークン制限の発生**: 巨大なファイルを読み書きする際、AIの出力限界に達しやすい。
- **副作用の増大**: UIの変更が、意図せず他のゲームモード（探索/合戦）のレイアウトに影響を及ぼしやすくなっている。

---

## 2. 根本対策：都の「遷宮」（モジュール化計画）

### A. App.jsx の軽量化
`App.jsx` は全体の「状態の統合」と「画面ルーティング」のみを担当するよう、役割を分散します。

| 現状の機能 | 移行先の新コンポーネント (src/components/...) |
| :--- | :--- |
| **ダイアログシステム (isStory含)** | `ui/DialogOverlay.jsx` |
| **合戦差し込み (Combat Interjection)** | `ui/CombatInterjection.jsx` |
| **メインUI (探索/合戦の窓)** | `layout/GameArea.jsx` |
| **デバッグパネル** | `debug/DebugPanel.jsx` |

### B. 常数（Constants）のさらなる分離
画像パスや装飾定数（`DIALOG_SPEAKERS` など）を、より細かい単位で `src/constants/` 下に小分けし、読み込み時間を短縮します。

---

## 3. 品質保証：都の「護法」（テスト戦略）

### A. ユニットテストの強化
`src/logic/` および `src/hooks/` に対するテストを拡充します。
- **方針**: 何か機能（例：新しい術の追加、ダメージ計算の変更）を追加する際、まず `*.test.js` を更新/実行し、論理が通ることを確認してから UI 実装に入る。

### B. チェックポイント制の導入
開発ステップごとに、以下の「検分の儀」を必須化します。
1. **静的検分 (Lint)**: 文字通り、都の理に反していないか確認。
2. **自動検分 (Auto-Dev)**: 特に UI 変更時は、ブラウザを通じた「視覚的検分」を自動で行い、PC/スマホ両面での無事を記録する。

---

## 4. 実行の順序（提案）
主殿のご承認をいただければ、以下の順で「再開発」を進めたいと存じます。

1. **第一期**: `activeDialog` (幽玄モード含む) を独立したコンポーネントへ。
2. **第二期**: メインUI（探索/合戦のコンテナ）を `App.jsx` から分離。
3. **第三期**: 開発フローに「テストファースト（まずVitestを実行）」を組み込む。
