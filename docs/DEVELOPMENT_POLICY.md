# ⛩️ 平安魔道伝 羅生門編：都の開発の方針 (DEVELOPMENT_POLICY.md)

都を不滅のものとするための、開発手法の憲法。

## 1. 言霊と理の分離（外部ファイル化）
- **掟**: バランス値、シナリオ、怪異のデータ、イベント配置などは、すべて `src/data/*.json` に外部化し、UIロジック（`App.jsx`）から分離する。
- **理由**: 都の調整（数値調整やテキスト変更）を容易にし、ロジックの改変による意図せぬ崩壊を防ぐため。
- **主な聖典（JSON）**:
    - `data/Balance.json`: 最大Lv、経験値テーブル、接敵率、ダメージ計算定数。
    - `data/Scenario.json`: オープニング、各ダイアログ、UIの呼称、ゲーム開始/終了時の言霊。
    - `data/Enemies.json`: 全怪異の御名、画像パス、ステータス、特殊フラグ。
    - `data/MapEvents.json`: 迷宮内の社・井戸・巻物の座標と、そこで綴られる言霊。

## 2. 羅生門への連撃（反復・MVP開発）
- **掟**: 一度に巨大な都を築こうとせず、常に「動く最小の都（MVP）」を維持しつつ、小分けに反復（イテレーション）して洗練を加える。
- **理由**: 理の不具合（バグ）を早期に発見し、主殿（ユーザー）に「常に触れられる状態」で検分いただくため。
- **手順**: 骨組み実装 → 意匠（UI）の注入 → 検分・浄化 のサイクルを一台の歯車のごとく回す。

## 3. 平安美学の死守（Premium UI / UX）
- **掟**: 都は美しく、奥行きがあり、漆黒の夜と黄金の月光のように、没入感を極限まで高めるべきである。
- **理由**: プレイヤーが真に「平安の闇」に立ち、その空気（UX）を感じるため。
- **基準**:
    - ブラウザ標準の無機質な部品を、可能な限り都の意匠（黄金、墨引き、紅、和紙の質感）で上書きすること。
    - 俗世の言葉（メタ表現/システム用語）を徹底的に排除し、世界観に即した言霊に置き換える。
    - 画像（怪異・武将・風景）は、常に筆致を感じさせる墨絵（sumi-e）スタイルで統一する。

## 4. 智の継続（ドキュメント駆動）
- **掟**: `PLANNING_LOG.md` および各仕様書（`DEBUG_SPEC.md`, `UI_STYLE_GUIDE.md`）を常に最新に保ち、開発方針をコードと同期させる。
- **理由**: AIアシスタントが交代（次セッション）しても、都の設計意図を正しく継承し、先祖返りや設計의ブレを死守するため。

## 5. 都の巡回検分（システム監査）
- **掟**: 大きな節目の際（リリースタグの打刻、大規模機能追加、システム全体の見直し時）には、必ず「システム監査（Audit）」を実施し、現状の理が聖典に準拠しているかを検分する。
- **掟**: 監査結果に基づき「改善計画（AUDIT_IMPROVEMENT_PLAN.md）」を作成し、改善の路（アクションプラン）を明確にした上で浄化に取り組むこと。
- **理由**: 開発の加速に伴い生じる「淀み（ハードコードや技術的負債）」を定期的に浄化し、都の不滅の品質を死守するため。

## 6. 都の再編と検分の理（モジュール化 ＆ テスト）
- **掟【分割の計】**: `App.jsx` が一定の規模（500行目安）を超えぬよう、機能ごとにコンポーネントや演算ロジックを分離せよ。1ファイルの情報量を抑えることでトークン制限を防ぎ、理の不一致を根絶する。抽出された部品は `src/components/` 配下の用途別ディレクトリ（`navigation/`, `status/`, `battle/` 等）へ遷座させること。
- **掟【検分の理】**: 論理の変更時は `Vitest` を用いた単体テストを伴うこと。`src/logic/` 配下の純粋演算は `npm run test` でその正しさを証明した上で、都へと統合する。
- **掟【双璧の検証】**: 変更の際は PC版（通常）と スマホ版（?mobile=1&debug=true）の両面を検分し、双方の調和（非退行）を確約せよ。

## 7. 遷座の儀（環境最適化）
- **掟**: PowerShell 環境では `&&` は不可。必ず **`;` (セミコロン)** でコマンドを連結せよ。重要節目ではリリース番号を更新し、GitHub へ功徳を記録すること。
