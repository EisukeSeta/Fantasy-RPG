# ⛩️ 羅生門：開発・統治規則（DEVELOPMENT_RULE）

本文書は、AIと人間が共創する「羅生門 RPG」における開発の「絶対法」を定めたものである。高品質・高保守性を維持するため、以下の手順を全ての開発において厳守せよ。

## 1. 開発の基本三原則（テストファースト）

1.  **設計の章（Design First）**: コードを一行たりとも書く前に、モジュールの「入出力」「変数定義」「責任範囲」を明文化せよ。
2.  **検証の章（Test First）**: 可能な限りテストコードを先に定義し、あるいはモジュール自体に自己診断機能を持たせよ。
3.  **監査の章（Audit Always）**: 既存の変数やロジックの破壊的変更は、理由の報告と主殿の承認なしには一切禁ずる。

## 2. AI の役割分立（Role Definition）

AI は状況に応じて以下の役割を使い分け、多角的な検分を行わなければならない。

*   **「設計者（Architect）」**: 
    - 実装前に `DATA_CONTRACTS.md` を参照し、データ構造への準拠を確認の上、設計 Artifact を提示する。
    - データ構造の変更がある場合は、必ず主殿の承認を仰ぐ。
*   **「実装者（Implementer）」**: 
    - 設計に基づき、副作用を抑えた純粋なコードを記述する。
    - **守備範囲の掟**: 一度の修正は原則一モジュール（＋そのテスト）に限定し、大規模な連鎖修正を禁ずる。
    - 全ての関数に JSDoc を付与し、型と意図を明示する。
*   **「監査官（Auditor）」**: 
    - 実装後、`npm run test` 等の自動検証、Build、Lint を実行する。
    - **照合の章（Diff Audit）**: 奉納（Git Push）前に必ず `git diff` を実行し、「不当なプロパティ消失がないか」「意図せぬコードの混入がないか」を一行ずつ検分する。
    - **監査レポート（Audit Report）**: 上記の検分結果（変更点の要約と安全性の証明）を主殿に報告し、承認を得る。
    - 必要に応じてブラウザエージェントによる視覚検証を実施し、証跡を残す。

*   **「迅速な理（Fast Fix）」**:
    - 単一ファイル内で 10 行以内の軽微な修正、または不具合（バグ）の緊急修復に限り、設計 Artifact の提示を省略し、即座に実装・報告することを認める。
    - ただし、実装後の `git diff` による自己検分と報告は、いかなる場合も省略してはならない。

## 3. モジュール設計の掟

1.  **単一責任（SRP）**: 一つのモジュールは一つの役割のみに集中せよ。長大なファイルは分割を検討せよ。
2. **不変性の尊重（Immutability of Design）**:
    - 既存の `GameContext`、`App.jsx`、データ・オブジェクト（Enemy, Player, Spell等）内の変数を削除・改名することを原則禁止する。
    - **削除禁止令**: 「不要に見える」というAI独自の判断でプロパティを削ってはならない。
    - 止むを得ず変更が必要な場合は、影響範囲の全リスト（削除チェックリスト）を提示し、主殿の承認を得よ。
3.  **コメントの理**: モジュール冒頭には必ず以下の情報をコメントとして残すこと。
    - 機能概要
    - 入力（Input Args / Context Value）
    - 出力（Return Value / State Updates）
    - 主要な内部変数

## 4. 意匠の掟 (UI Consistency)
1. **統合美の尊厳**: 全ての UI 追加・変更は、**`docs/UI_DESIGN_GUIDELINE.md`** に定められた理に従い、レイアウト、フォント、配色の一貫性を維持せよ。
2. **揺れの禁止**: 画面間の遷移（捷径の巡回等）において、ヘッダー、タイトル、閉じボタンの配置を 1px 単位で揃え、ユーザーに「視覚的な揺れ」による違和感を与えてはならない。
3. **部品の再利用**: 個別のインラインスタイルを乱発せず、`src/index.css` に定義された共通クラス（`.yugen-modal-*` 等）を優先的に活用せよ。

## 5. 検証の掟

1.  **ユニットテスト**: ロジック（計算・移動等）の変更時は、テストデータを用いて期待通りの結果（アウトプット）になることを証明せよ。
2.  **ブラウザ検分**: UI の変更時は、PC版・スマホ版の双方で「ビジュアルの変化がないこと」あるいは「意図通りであること」を証跡（PNG）をもって示せ。

## 6. 執行環境と効率の掟

1.  **コマンドの理 (PowerShell Compliance)**:
    - AI が実行する全てのコマンドは **Windows PowerShell** 環境を前提とする。
    - 複数コマンドの連結には、Unix 系の `&&` ではなく、必ず **`;` (セミコロン)** を使用せよ。
    - パス区切り文字には、原則として `/` ではなく `\` または環境に合わせた正しい形式を用いよ。
2.  **効率の理 (Performance & Optimization)**:
    - モジュール設計時は、React の再レンダリングを最小限に抑えるよう、メモ化（`useMemo`, `useCallback`）とコンポーネントの細分化を徹底せよ。
    - 特に `GameContext` や巨大な Hook（`useCombat` 等）の依存関係による「連鎖的な再描画」に常に留意し、必要に応じてベンチマークによる KPI 計測（レンダリング回数等）を実施せよ。
    - **[行動予約（絶縁ガード）の理]**: 非同期のオートバトル AI や連打防止が必要な操作においては、「行動予約 (Processed Turn ID) システム」を用いる。Ref と State のターン ID を照合することで、React の再レンダリングや非同期タイマーによる「二重発火（連打バグ）」を物理的に防止せよ。
