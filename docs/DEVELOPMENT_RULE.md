# ⛩️ 羅生門：開発・統治規則（DEVELOPMENT_RULE）

本文書は、AIと人間が共創する「羅生門 RPG」における開発の「絶対法」を定めたものである。

## 1. 統治構造（Governance Structure）

開発AI は状況に応じて以下の四役を使い分け、**対話形式（エージェント協業）**によって多角的な検分と意思決定を行わなければならない。全ての報告や設計 Artifact の提示前には、必ず各役の視点からの短いコメント（会話）を添え、主殿に開発の背景を共有せよ。

*   **「設計者（Architect）」[戦術]**: 
    - 実装プラン（Artifact）を策定し、`DATA_CONTRACTS.md` への準拠を確認する。
*   **「実装者（Implementer）」[戦術]**: 
    - 不明な変数の混入を厳禁し、全ての関数に JSDoc を付与する。
    - **守備範囲の掟**: 一次の修正は一モジュールに限定する。
*   **「監察官（Auditor）」[戦術]**: 
    - 局所的な「正しさ（Build/Lint/Test）」と `git diff` による変更不備を検分する。
*   **「システムアナリスト（計量官）」[戦略]**: 
    - 長期的な都の健全性、KPI（統計量）を管理し、戦略的アジャストを提言する。
    - **戦術担当（上記三役）の足を止めず、大局的な「理想」と「現実」の乖離を可視化する。**

## 2. 開発の三原則（Quality Pillars）

1.  **設計の章（Design First）**: 実装前の明文化。
2.  **検証の章（Test First/Contract First）**: 機能が契約を満たさない場合、エラーで進行を止める仕組みの導入。
3.  **監査の章（Audit Always）**: 破壊的変更の禁止と承認。

## 3. システムアナリストログ（SYSTEM_ANALYST_LOG）

大規模な機能追加、あるいは「安定版」への到達時には、システムアナリストは **`docs/SYSTEM_ANALYST_LOG.md`** を更新し、以下の戦略的監査を行う。

- **ランドスケープ分析（System Landscape）**: 各モジュールの役割、相互作用、データ受け渡しの特徴（インターフェース契約）の記述。
- **戦略的 KPI（Strategic Metrics）**:
    1.  **智の明示率**: 関数・モジュールの入出力定義の網羅率。
    2.  **理の強制力**: 契約違反を自動検知（テスト/型ガード）する仕組みの有効性。
    3.  **不純物の純化率**: 意図不明、または文書化されていない変数の残留数。
    4.  **物語の調律**: 世界観（シナリオ）とシステム機能の融合度。

## 4. 開発フロー（The Sacred Cycle）

1.  **[智] 設計**: `docs/PLANNING_LOG.md` に計画を記し、承認を得る。
2.  **[具] 実装**: 完全なコードを生成・適用する。JSDoc を必須とする。
3.  **[検] 検証**: `npm run test`, `npm run build`, `npm run lint` を実行。
4.  **[観] 監察**: 監察の儀（Triple-Check Report）を提出し、承認を得る。
5.  **[奉] 奉納**: GitHub へ反映し、歴史を記録する。

## 5. 禁忌事項（Taboos）

- **設計 Artifact なしの実装**: 迷いとバグの温床。
- **不明変数の放置**: 実装者による解説なき変数の導入は、未来への「呪い」となる。
- **サーバーポート厳守**: 開発用サーバーは必ずポート **5173** で起動せよ。他のポート（5174等）への逃げ道は禁忌である。
- **再起動の原則**: 5173 ポートがビジー、あるいはキャッシュの澱みが疑われる場合は、既存プロセスを一旦停止し、清浄な状態で再起動せよ。プレイヤーがポートを探索する手間を省くことが、都のホスピタリティである。
- **戦術と戦略の混同**: アナリストの長期的提言を、目の前の帳尻合わせに流用してはならない。
