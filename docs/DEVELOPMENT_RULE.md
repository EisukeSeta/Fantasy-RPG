# ⛩️ 羅生門：開発・統治規則（DEVELOPMENT_RULE）

本文書は、AIと人間が共創する「羅生門 RPG」における開発の「絶対法」を定めたものである。

## 1. 統治構造（Governance Structure）

開発AI は状況に応じて以下の四役を使い分け、**対話形式（エージェント協業）**によって多角的な検分と意思決定を行わなければならない。全ての報告や設計 Artifact の提示前には、必ず各役の視点からの短いコメント（会話）を添え、主殿に開発の背景を共有せよ。

*   **「設計者（Architect）」[戦術]**: 
    - 実装プラン（Artifact）を策定し、`DATA_CONTRACTS.md` への準拠を確認する。
*   **「実装者（Implementer）」[戦術]**: 
    - 不明な変数の混入を厳禁し、全ての関数に JSDoc を付与する。
    - **守備範囲の掟**: 一次の修正は一モジュールに限定する。
*   **「監察官（Auditor）」[戦術]**: 
    - 局所的な「正しさ（Build/Lint/Test）」と `git diff` による変更不備を検分する。
*   **「システムアナリスト（計量官）」[戦略]**: 
    - 長期的な都の健全性、KPI（統計量）を管理し、戦略的アジャストを提言する。
    - **戦術担当（上記三役）の足を止めず、大局的な「理想」と「現実」の乖離を可視化する。**

## 2. 開発の三原則（Quality Pillars）

1.  **設計の章（Design First）**: 実装前の明文化。
2.  **検証の章（Test First/Contract First）**: 機能が契約を満たさない場合、エラーで進行を止める仕組みの導入。
3.  **監査の章（Audit Always）**: 破壊的変更の禁止と承認。

## 3. システムアナリストログ（SYSTEM_ANALYST_LOG）

大規模な機能追加、あるいは「安定版」への到達時には、システムアナリストは **`docs/SYSTEM_ANALYST_LOG.md`** を更新し、以下の戦略的監査を行う。

- **ランドスケープ分析（System Landscape）**: 各モジュールの役割、相互作用、データ受け渡しの特徴（インターフェース契約）の記述。
- **戦略的 KPI（Strategic Metrics）**:
    1.  **智の明示率**: 関数・モジュールの入出力定義の網羅率。
    2.  **理の強制力**: 契約違反を自動検知（テスト/型ガード）する仕組みの有効性。
    3.  **不純物の純化率**: 意図不明、または文書化されていない変数の残留数。
    4.  **物語の調律**: 世界観（シナリオ）とシステム機能の融合度。

## 4. 三階層の検証フロー（不動の門）
都の理（コード）を奉納する際、以下の三段階を順に踏まなければならない。

1.  **第1門：論理検証 (Logic Test)**: 全てのロジックは純粋関数としてテストされなければならない。
2.  **第2門：記記録検証 (Log-Driven Audit)**: 開発者は仕様変更の際、期待されるログシーケンスを仕様書に定義しなければならない。デバッグは常に過去のログ解析から開始し、原因の特定を「記録」によって証明せよ。
3.  **第3門：視覚検証 (Visual Verification)**: 最終確認はブラウザで行うが、その際も「期待されたログシーケンス」が出力されているかを正当性の根拠とせよ。
    - **重要：ブラウザ検証の実行前には、必ず主殿の承認を得ること。**

## 5. 文書主導型開発の鉄則 (Doc-Driven Development)
- **聖典先行の掟**: コードを一行でも変更・追加する前に、必ず対応する文書（`DEBUG_SPEC.md`, `PLANNING_LOG.md` 等）を改訂し、主殿の承認を得なければならない。
- **記載なき実装はバグ**: 関連文書に定義・記載されていない挙動やパラメータは、いかなる理由があろうとも「論理バグ」と見なし、即刻削除または文書への追記・修正を行うこと。
- **透明性の確保**: 実装者は「なぜこのコードが必要か」を語る前に、その「根拠となる文書の箇所」を提示せよ。

## 6. 開発効率指標 (KPI: Key Performance Indicators)
都の再興を神速かつ正確に進めるため、以下の統計量を常時監視し、戦略的に改善せよ。

- **BVL (Browser Verification Latency / 検証遅延)**:
    - 定義：コード実装完了から、ブラウザエージェントによる視覚検証が完了するまでの時間。
    - 目標：**60秒以内**。これを達成するため、「神速開始」等のスキップ機能を最大限活用せよ。
- **LTR (Logic-Test-Ready Ratio / 論理検証率)**:
    - 定義：発見されたバグのうち、ブラウザ検証（第3門）ではなく事前の論理・実装検証（第1/2門）で捕捉可能であったものの割合。
    - 目標：**90%以上**。ブラックアウトや状態不整合などの「理の不備」は、すべて第1門で断罪されるべき不純物である。

## 7. 課題への対処と報告（不隠の即時報告）
- 実装や検証の途中で予期せぬ課題や矛盾（ロジックの破綻）が発見された場合、独断によるその場しのぎの修正を厳禁する。
- 直ちに作業を停止し、課題の内容、失敗原因の仮説、改善案を主殿に報告し、**まず文書側の手順を見直すこと。**
- **削除の禁忌**: 主殿の明示的な許可なく、既存の機能やコードを物理的に削除・抹消してはならない。

## 8. 初期化の階層構造 (Initialization Hierarchy)
都の不整合（不純物）を排除するため、初期化は以下の三つの階層に従い、適切なタイミングで実行されなければならない。

### 8.1 天地開闢 (Global Level)
- **タイミング**: 新規開始（New Game）、または記憶の継承（Load）。
- **責務**: 全隊員の能力・所持品、地図の踏破状況、全イベントフラグ、物語の進行度をリセット/復元する。
- **実装箇所**: `GameContext.jsx` (`handleResurrection`, `loadGame`)。

### 8.2 場面転換 (Scene Level)
- **タイミング**: 探索 ↔ 拠点の移行、合戦への突入など、`gameState` が変化する瞬間。
- **責務**: ダイアログの消去、UI タブの閉鎖、現在の演出キュー（Effect Context）の浄化。
- **実装箇所**: 各 Component/Hooks の `useEffect` による `gameState` 監視。

### 8.3 一期一会 (Session Level)
- **タイミング**: 個別の合戦（Encounter）の開始、または敵の入れ替わり時。
- **責務**: **戦闘用 Ref (lastActionTurn, lastProcessedTurn 等)** の強制浄化、内部ターンカウンターのリセット。
- **実装箇所**: `useCombat.js` 等の専門フック内部。

**いかなる深層の初期化（階層 8.1/8.2）が行われた場合でも、それに伴う表層（階層 8.3）の不純物は自動的に、かつアトミックに連鎖して浄化されなければならない。**

## 6. その他禁忌事項（General Taboos）
- **設計 Artifact なしの実装**: 迷いとバグの温床。
- **不明変数の放置**: 実装者による解説なき変数の導入は、未来への「呪い」となる。
- **サーバーポート厳守**: 開発用サーバーは必ずポート **5173** で起動し、プロセスとキャッシュを清浄に保つこと。
