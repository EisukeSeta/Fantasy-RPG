# 📜 開発拠点再編計画：平安魔道伝「品質の理」

本文書は、プロジェクトの規模拡大に伴う「トークン制限」および「デバッグの複雑化」を解決するための、根本的な構造改革案である。

## 1. 現状の課題
- **App.jsx の肥大化**: 600行を超え、AIが全容を把握するのに多くのトークンを消費する。
- **後退バグのリスク**: 戦闘演出を追加するとマップが壊れるなど、機能間の依存が不透明。
- **デプロイの不安定性**: 環境依存のコマンドエラーや、スマホ版の確認漏れが発生。

## 2. 解決策：三位一体の再編

### ① 【理の抽出】ロジックと定数のモジュール化
`App.jsx` を以下の役割に分解する。

| 分類 | ファイル名（予定） | 内容 |
| :--- | :--- | :--- |
| **定数** | `src/constants/gameData.js` | 画像パス、MAPサイズ、DIRECTIONS、経験値テーブル。 |
| **演算** | `src/logic/combat.js` | 命中判定、ダメージ計算、痛打判定。 |
| **演算** | `src/logic/exploration.js` | 移動判定、周囲8マス視認ロジック。 |
| **部品** | `src/components/battle/` | 敵アイコン、浮遊数字、勝利演出。 |
| **部品** | `src/components/status/` | 隊員証、HP/MPバー。 |
| **部品** | `src/components/labyrinth/` | 3Dビュー、地図レンダリング、ログ。 |

### ② 【検分の自動化】テストファーストの導入
`Vitest` を導入し、機能変更のたびに「理（正解）」を自動確認する。

- **重点テスト項目**:
    - **戦闘**: レベル差による命中率の変化が期待通りか。
    - **成長**: レベル50到達時にステータスが破綻していないか。
    - **探索**: 特定の座標でイベントが発火するか。

### ③ 【聖典の進化】auto-dev.md 統合フロー
```markdown
1. [検証] npm run test  (既存機能の破壊がないか確認)
2. [実装] 小規模なモジュール単位でのコード適用
3. [検分] npm run build ＆ PC/Mobile 両面ブラウザ検証
4. [昇格] バージョン番号のインクリメント
5. [遷座] Git操作 (; 連結による PowerShell 最適化実行)
```

## 3. 実装の進め方
1. **フェーズ1**: `Vitest` の導入と、最も重要な `combat.js` の抽出（テストコード完備）。
2. **フェーズ2**: 定数データ（画像や設定値）の外部化。
3. **フェーズ3**: UIコンポーネントの順次分割（StatusPane -> EnemyPane -> LogPane）。

---
**Senior Frontend Engineer の見解**: 
この再編により、1回あたりのコード修正範囲が限定され、AIの精度が飛躍的に向上します。また、テストコードが「仕様書」として機能するため、将来のバランス調整が非常に安全になります。
