# アルゴリズムレビュー：鵺戦後ハングアップの浄化

## 1. 現状の論理欠陥（As-Is / stable との差分）
1. **[起]**: 「鵺」のHPが0になり、合戦が終了（finalizeBattle 実行）。
2. **[承]**: `useNavigation.js` のボス遭遇判定（`nX === BOSS_POS.x`）は、**「移動時（processMove）」**にしか発火しない設計。
3. **[転]**: 戦闘終了直後、プレイヤーは座標(x,y)に留まったままであり、「移動」がないため凱旋ダイアログのトリガーが引かれない。
4. **[結]**: `DialogManager` は前回の「次第を追う」という残滓を表示したまま、実体のない物語を待機し続け、ハングアップする。

## 2. 解決のアルゴリズム案（To-Be）
「移動」をトリガーにするのをやめ、**「状態の変化（bossDefeated）」をトリガーにする監視の理**を導入する。

### 🔄 状態遷移の論理フロー
1. **[合戦終了（勝利）]**:
   - `finalizeBattle` が呼ばれる。
   - `bossDefeated` フラグが `true` に更新される。
2. **[理の監視者（Watcher）]**:
   - `App.jsx` または `useNavigation` 側で、`bossDefeated` の変化を監視する。
   - 「ボスが今倒された」かつ「現在地がボス座標」であることを検知する。
3. **[凱旋の幕開け]**:
   - 移動を待たず、即座に「凱旋ダイアログ」を `setActiveDialog` へ流し込む。
4. **[幕引きと還流]**:
   - ダイアログの `onConfirm` にて `setActiveDialog(null)` と `setGameState('EXPLORING')` を確実に実行し、探索モードへ戻す。

---
主殿、この「移動を待たず、状態変化で凱旋を呼ぶ」という論理フロー、合意を頂けますでしょうか。
合意頂いたのち、この論理を最短かつ堅牢に実現するための「 micro-diff（二つの一筆）」を提示いたします。
