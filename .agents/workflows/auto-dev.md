1. **[調査] 記記録（ログ）的解析と開発会議**
    - `docs/PLANNING_LOG.md` および過去のブラウザエージェントが残した **「記記録（Logger 出力）」** を最優先で解析する。
    - **ログ駆動の現状分析**: 新たに実機を動かす前に、既存の記録から「理の逸脱（不整合）」の座標を特定せよ。
    - 都の聖典（`DEBUG_SPEC.md`, `LOGGING_RULE.md` 等）を読み込み、現状を同期する。
    - **エージェント協業**: 調査結果（ログ解析）を受け、四役による分析を表明し、主殿に状況を報告する。
2. **[設計] 智の構築（聖典先行）**
    - タスクの規模に応じ、実装プランを `docs/PLANNING_LOG.md` に記す。
    - **契約仕様の法（Contract First）**: モジュール間（Context等）の接続情報を変更・追加する場合、必ずコード変更前に `docs/CONTRACT_SPEC.json` を改訂せよ。「文書にない情報は、この都に存在せぬ」と心得よ。
    - **四役事前合意**: 修正方針は法典に基づき四役（戦略・戦術・監査・実装）の合議を経て、主殿の承認を得ること。
    - **文書主導の鉄則**: コード変更前に、必ず対応する文書（仕様書・期待されるログシーケンス等）を改訂すること。
    - 段階的実装の掟: 大規模な機能追加時は TODO リストを作成し、各工程を主殿に確認・報告しながら着実に行う。
3. **[実装] 具現化**
    - 聖典（ルール）に従い、一切の省略なき完全なコードを生成・適用する。
    - **守備範囲の掟**: 一度の修正は原則一モジュール（＋そのテスト）に限定せよ。
4. **監査と検証 (Audit & Verify)**:
    - **第1門：論理検証 (Logic Test)**: `docs/DEVELOPMENT_RULE.md` に従い、副作用のない「純粋関数」として論理の正しさを 100% 証明せよ。
    - **第2門：実装検証 (Implementation Test)**: 統合されたモジュールやフックの状態遷移、および `npm run test`, `lint`, `build` の合格を確認せよ。
    - **第3門：視覚検証 (Visual Verification)**: **（主殿の承認必須）** ブラウザエージェントを用い、実機での表示と機能を最終確認せよ。
        - **掟：三度の限度**: 自律的なリトライ（派遣）は【最大3回】まで。
        - **掟：十秒の沈黙**: 操作後【10秒間】無反応な場合はハングアップと断定し、即座に終了せよ。
        - **掟：論理の証明**: 失敗・停滞時は、必ずソースコードの依存関係やガード条件から「なぜ動かぬか（論理的推論）」を詳述して日本語で報告せよ。
5. **[照合] 奉納前の儀式 (Diff Audit)**
    - **git diff の実行**: 奉納前に必ず `git diff` および `git diff --staged` を実行し、変更点を詳細に読み取ること。
    - **監査報告**: 「削除されたプロパティの有無」「意図せぬコード混入がないこと」を主殿に報告し、承認を得ること。
6. **[完了] 遷座と記録**
    - **Git操作（理）**: PowerShell環境では `;`（セミコロン）でコマンドを連結。
    - 承認後、GitHub へ反映し、開発史を `docs/PLANNING_LOG.md` に要約する。
